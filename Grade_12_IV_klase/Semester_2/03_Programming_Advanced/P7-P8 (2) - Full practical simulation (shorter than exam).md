---

## Context

End-to-end build under time constraints; strict marking.

## Completion Definition

Student independently completes a full programming practical simulation (shorter than exam) that requires:

- Reading and interpreting a specification correctly

- File input + parsing (txt/csv) per spec

- Correct use of composite structures (records + collection)

- Correct key-based operations (e.g., select/search/sort/modify as required)

- Correct, specification-compliant output formatting

Execution must be correct under timed conditions and represent an end-to-end rehearsal of exam-like workflow.

---

## Minimum Evidence

Student submits within time limit:

- Complete working program

- Input file used

- Produced output (exact format)

- Teacher-logged completion time

Constraints:

- No scaffolding or step-by-step prompts

- No mid-run teacher correction

---

## Cognitive Level (Bloom's)

**Bloom: Apply → Analyze**

- Apply full-stack programming mechanics under pressure

- Analyze task requirements to choose correct operations and sequencing

---

## Readiness Gate

✔ All must be true:

- Program runs without error on valid input

- Parsing correct (record count and field types correct)

- Required operations completed correctly (per spec)

- Output formatting exact (no extra text)

- Completed within defined simulation time limit

If any required component fails or time exceeded → Gate not met.

---

## Common Failure Modes

- Misreading spec constraints → wrong operation chosen

- Parsing errors propagating into wrong results

- Composite structure mis-modeled (wrong field types/order)

- Key logic errors (wrong field used for comparison)

- Output format drift under time pressure

- Not handling empty/not-found cases safely

- Spending too long debugging trivial formatting
