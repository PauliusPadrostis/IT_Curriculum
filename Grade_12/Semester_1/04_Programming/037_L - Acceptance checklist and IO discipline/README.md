## Overview

- Minimal acceptance checklist that every solution must pass before submission
- Specification reading and IO discipline as a unified "before you code" framework

## Sub-topics

- Building a personal acceptance checklist: compiles, runs, correct I/O, edge cases covered, no debug output
- Systematic specification reading: identifying all requirements before writing code
- IO discipline: input consumption order, output format compliance, end-of-input handling
- Using the checklist as a final gate before submission

---

## Completion Definition

Student can apply a structured acceptance checklist to any solution AND demonstrate disciplined specification reading and IO handling as a pre-coding routine.

## Minimum Evidence

- Produces and applies a personal acceptance checklist to a given solution
- Demonstrates systematic spec reading: extracts all IO requirements before coding
- Identifies IO discipline violations in sample programs (wrong read order, extra output, missing end-of-input handling)

## Bloom's Taxonomy Level

Level 3 — Apply

## Readiness Gate

All must be true:

- Input parsing structure matches spec exactly
- Output formatting matches required format exactly (spacing, order, precision)
- Program runs without runtime error on sample input
- All constraints identified correctly in spec annotation
- Checklist applied and all items verified before submission

If any formatting mismatch or misread constraint -> Gate not met.

---

## Common Failure Modes

- Skipping constraint reading (e.g., max N)
- Ignoring edge cases mentioned in spec
- Wrong delimiter handling
- Incorrect output spacing or line breaks
- Printing extra debug text
- Assuming input format instead of verifying
- Submitting without running the acceptance checklist
- Reading input in wrong order or missing trailing input
